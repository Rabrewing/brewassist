# S4.10c.4 DevOps 8 Implementation

## DevOps 8 Signal Contract v1

This document defines the canonical telemetry schema for DevOps 8 signals, ensuring structured truth for panes and tabs.

### Signal Schema

Each DevOps signal must include:

- `id`: Unique identifier (string from predefined enum)
- `label`: Human-readable label
- `status`: One of "optimal", "degraded", "stalled", "unknown"
- `value`: Numeric score (0-100)
- `source`: Source system (e.g., "execution_planner")
- `timestamp`: ISO timestamp string
- `confidence`: Confidence level (0.0-1.0)
- `notes`: Short explanatory text

### Admin Debug Info

Signals may include `adminDebugInfo` object with additional metadata, but this is stripped in Customer Mode to prevent leaks.

### Normalization Rules

- Status is normalized to valid enum values
- Value is clamped to 0-100
- Confidence is clamped to 0.0-1.0
- Required fields are enforced

### Signal Registry + Adapters

The DevOps 8 signals are computed using an adapter pattern:

- Each signal has a dedicated adapter function in `lib/devops8/adapters/`
- Adapters accept context objects and return normalized signals
- The registry in `lib/devops8/registry.ts` defines visibility and compute functions

### Signal IDs

The 8 DevOps principles:

1. flow_integrity: Flow Integrity
2. feedback_velocity: Feedback Velocity
3. learning_memory_integrity: Learning & Memory Integrity
4. build_change_quality: Build & Change Quality
5. scope_containment: Scope Containment
6. safety_policy_enforcement: Safety Policy Enforcement
7. reasoning_visibility: Reasoning Visibility
8. execution_efficiency: Execution Efficiency
