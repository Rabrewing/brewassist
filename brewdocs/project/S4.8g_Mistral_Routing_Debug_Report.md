**BrewAssist S4.8g (Mistral Routing & Verification) Debug Report**

**Date:** December 9, 2025

**Current Status:**
S4.8g implementation is in progress. Acceptance tests are failing.

**Overall Goal:**
Enhance BrewAssist UI/UX, integrate persona/memory, and establish a multi-model AI chain with dynamic routing (including NIMs research, Mistral verification), BrewTruth grading, and robust fallback mechanisms.

**Problem Description:**
The primary issues are related to the `resolveRoute` function in `lib/model-router.ts` not correctly prioritizing preferred providers (Mistral) and the `runBrewAssistEngine`'s fallback logic in `lib/brewassist-engine.ts` not behaving as expected, particularly regarding `routeType` assignment and unnecessary subsequent calls to other providers after a successful fallback.

**Detailed Diagnosis:**

1.  **`resolveRoute` Function (Test 5 Failure):**
    *   **Expected Behavior:** When `preferredProvider: "mistral"` is passed, `resolveRoute` should return a route configured for Mistral.
    *   **Observed Behavior:** `resolveRoute` is consistently returning the default primary route (OpenAI) for the `llm` role, even when `preferredProvider: "mistral"` is present in the `opts`.
    *   **Debugging Output (`console.log` from `lib/model-router.ts`):**
        ```
        resolveRoute - role: llm opts: { useResearchModel: false, preferredProvider: 'mistral' }
        resolveRoute - preferredProviderConfig: {
          enabled: false,
          baseUrl: 'https://api.mistral.ai/v1',
          apiKey: undefined,
          primaryModel: 'mistral-small-latest',
          secondaryModel: 'mistral-large-latest'
        }
        resolveRoute - Returning default primary route: { provider: 'openai', model: 'gpt-4.1-mini', routeType: 'primary' }
        ```
    *   **Root Cause:** The `preferredProviderConfig` for Mistral shows `enabled: false` and `apiKey: undefined`. This is the critical finding. Although `MISTRAL_ENABLED: "true"` and `MISTRAL_API_KEY: "test-mistral-key"` are set in `MOCK_ENV` within `__tests__/api/brewassist.test.ts`, these environment variables are *not* being correctly picked up by the `MODEL_PROVIDERS` object in `lib/model-router.ts` when the module is loaded. This suggests a module loading/environment variable propagation issue in the test environment. `MODEL_PROVIDERS` is initialized at module load time, and if `process.env` is mocked *after* `lib/model-router.ts` has been imported, `MODEL_PROVIDERS` will not reflect the mocked values.

2.  **`runBrewAssistEngine` Fallback Logic (Test 6 Failure):**
    *   **Expected Behavior:** When `preferredProvider: "mistral"` fails and falls back to OpenAI, the engine should return the OpenAI response with `routeType: "fallback"` and stop further processing.
    *   **Observed Behavior:**
        *   Mistral fails (as mocked).
        *   OpenAI is called as a fallback and succeeds (as mocked).
        *   However, the engine then proceeds to call Gemini, which is unexpected.
        *   The `routeType` returned is `"primary"` instead of `"fallback"`.
    *   **Debugging Output (`console.log` from `__tests__/api/brewassist.test.ts` and `lib/model-router.ts`):**
        ```
        createMockFetch - Fetching URL: https://api.mistral.ai/v1/chat/completions
        createMockFetch - Fetching URL: https://api.openai.com/v1/chat/completions
        createMockFetch - Fetching URL: https://generativelanguage.googleapis.com/v1beta/models // Unexpected call
        ```
    *   **Root Cause:** The `runBrewAssistEngine` function's `try...catch` block for handling provider failures and subsequent fallback attempts is not correctly terminating after a successful fallback. It appears to be iterating through all possible fallbacks defined in `MODEL_ROUTES` even after a successful response. Additionally, the `routeType` is not being updated to `"fallback"` when a fallback provider is successfully used.

**Actions Taken So Far:**

1.  **Temporarily disabled NIMs-related tests:** Commented out Test 1-4 in `__tests__/api/brewassist.test.ts` to focus on Mistral debugging.
2.  **Refined `BrewProviderConfig` interface:** Added `routeType?: 'primary' | 'fallback' | 'research' | 'preferred';` to `lib/model-router.ts` to allow for more granular route type assignment.
3.  **Reordered `MODEL_PROVIDERS` and `MODEL_ROUTES` definitions:** Moved them above the `pickNimsModel` import in `lib/model-router.ts` to address potential module loading issues (this resolved a `ReferenceError`).
4.  **Created `createMockFetch` helper:** Implemented a flexible `createMockFetch` function in `__tests__/api/brewassist.test.ts` to manage `global.fetch` mocks more effectively on a per-test basis.
5.  **Updated all relevant tests to use `createMockFetch`:** Replaced direct `global.fetch` assignments with calls to `createMockFetch` in Test 5, 6, 7, and the initial primary provider test.
6.  **Added extensive `console.log` statements:** Inserted `console.log`s in `createMockFetch` and `resolveRoute` to trace execution flow and variable values.
7.  **Prioritized `preferredProvider` in `resolveRoute`:** Reordered the logic in `resolveRoute` to check for `preferredProvider` before `useResearchModel` and the default primary route.

**Next Steps (Proposed Resolution Plan):**

1.  **Address Environment Variable Propagation for `MODEL_PROVIDERS`:**
    *   **Hypothesis:** `MODEL_PROVIDERS` in `lib/model-router.ts` is being initialized *before* `process.env` is mocked in `__tests__/api/brewassist.test.ts`.
    *   **Solution:**
        *   Modify `MODEL_PROVIDERS` to be a function that reads `process.env` *when called*, rather than at module load time. This would ensure it always gets the latest mocked environment variables.
        *   Alternatively, ensure `setupEnv()` is called *before* `lib/model-router.ts` is imported in the test environment, which might be tricky with Jest's module loading. The function approach is more robust.
        *   **Action:** Implement `getMODEL_PROVIDERS()` function in `lib/model-router.ts` and update `resolveRoute` to use it.

2.  **Refine `runBrewAssistEngine` Fallback Logic:**
    *   **Hypothesis:** The `runBrewAssistEngine` is not correctly breaking out of its fallback loop after a successful response.
    *   **Solution:**
        *   Review the `try...catch` block and the loop structure within `runBrewAssistEngine` to ensure it explicitly returns or breaks once a successful response is received from any provider (primary, preferred, or fallback).
        *   Ensure `routeType` is correctly set to `"fallback"` when a fallback provider is successfully used.
        *   **Action:** Inspect and modify `runBrewAssistEngine` in `lib/brewassist-engine.ts` to implement correct fallback termination and `routeType` assignment.

3.  **Re-enable NIMs Tests (After Mistral Fixes):** Once Mistral routing and fallback are verified, re-enable the NIMs tests and ensure they still pass with the updated `resolveRoute` logic.
