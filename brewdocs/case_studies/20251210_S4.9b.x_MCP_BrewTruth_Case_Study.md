#### 3.2. BrewTruth API Wiring and Environment Variable Debugging

**Core Challenge: `BREWTRUTH_ENABLED` Environment Variable Loading**

**Problem Description:**
Despite `BREWTRUTH_ENABLED=true` being present in `.env.local` and explicitly added to the `env` object in `next.config.js`, `process.env.BREWTRUTH_ENABLED` was consistently `undefined` inside the Next.js application's server-side code (specifically `lib/brewassist-engine.ts`) during development (`pnpm dev`). This prevented the `BrewTruth` grading block from executing, causing `truth: null` in API responses.

**Debugging Steps & Hypotheses:**
1.  **Initial Hypothesis:** Environment variable not set.
    *   **Action:** Attempted `export BREWTRUTH_ENABLED=true` (failed to persist across server restarts). Attempted `write_file` to `.env.local` (cancelled by user, but content was confirmed present).
    *   **Finding:** User confirmed `.env.local` contained `BREWTRUTH_ENABLED=true`.
2.  **Hypothesis:** Next.js not loading `.env.local` or `next.config.js` `env` object correctly.
    *   **Action:** Added `console.log("BREWTRUTH_ENABLED:", process.env.BREWTRUTH_ENABLED);` in `lib/brewassist-engine.ts`.
    *   **Finding:** Console output showed `BREWTRUTH_ENABLED: undefined`.
3.  **Hypothesis:** Next.js/Turbopack specific issue with server-side environment variables in development.
    *   **Action:** Performed web search for "Next.js environment variables server-side standalone turbopack".
    *   **Finding:** Confirmed known issues with environment variable handling in Next.js with Turbopack, especially in development.

**Proposed Temporary Hack (and user's decision):**
*   **Proposal:** Modify `lib/brewassist-engine.ts` to `const BREWTRUTH_ENABLED = process.env.BREWTRUTH_ENABLED === "true" || process.env.NODE_ENV === "development";`
*   **Rationale:** This would force `BrewTruth` to be enabled in development, bypassing the environment variable loading issue for testing.
*   **User Decision:** User (after consulting ChatG) approved the hack as a temporary measure to "keep moving," but emphasized the need to "harden" the solution later.

**Resolution for Development Environment (Hardening):**
*   **Action:** Instead of the `NODE_ENV` hack, the `dev` script in `package.json` was modified to explicitly set the environment variable: `"dev": "BREWTRUTH_ENABLED=true next dev -p 3000"`.
*   **Verification (Curl Test 2 - `/api/brewassist` with `BREWTRUTH_ENABLED=true`):**
    *   Initial runs failed with `truth: null`.
    *   After `package.json` modification and `pnpm dev` restart, **Result:** TEST 2 passed, returning a non-null `truth` object with `truth.version = "2"`.
*   **Verification (Curl Test 3 - `/api/brewassist` with `BREWTRUTH_ENABLED=false`):**
    *   Modified `package.json` to `"dev": "BREWTRUTH_ENABLED=false next dev -p 3000"`.
    *   After `pnpm dev` restart, **Result:** TEST 3 passed, returning `truth: null`.

**Jest Test for `BREWTRUTH_ENABLED` Logic:**

**Actions:**
*   Created `__tests__/api/brewtruth.test.ts` to test the conditional logic of `BREWTRUTH_ENABLED` in `lib/brewassist-engine.ts`.
*   Mocked external dependencies (`model-router`, `brewtruth`, `node-fetch`) to isolate the test.
*   Tested three scenarios: `BREWTRUTH_ENABLED="true"`, `BREWTRUTH_ENABLED="false"`, and `BREWTRUTH_ENABLED=undefined`.
*   Modified `lib/brewassist-engine.ts` to initialize `truth` to `null` to align with test expectations (`expect(result.truth).toBeNull()`).

**Challenges:**
*   Initial Jest test failures due to `result.truth` being `undefined` instead of `null` when `BrewTruth` was disabled.

**Resolution:**
*   Initialized `truth: BrewTruthReport | null = null;` in `lib/brewassist-engine.ts`.

**Verification:**
*   **Result:** All three Jest tests passed, confirming the logic for `BREWTRUTH_ENABLED` is sound.

### 4. Lessons Learned

*   **Environment Variable Nuances:** Next.js environment variable loading, especially with `output: 'standalone'` and Turbopack in development, can be highly unpredictable. Relying solely on `.env.local` or `next.config.js` for server-side variables in development might not be sufficient.
*   **Direct Script Modification:** For critical environment variables in development, explicitly setting them in the `package.json` script (e.g., `BREWTRUTH_ENABLED=true next dev`) proved to be the most reliable method to ensure they are picked up by the Next.js process.
*   **Importance of Targeted Tests:** The Jest tests were crucial in isolating the logic of `BREWTRUTH_ENABLED` from the environment loading issue, allowing us to confirm the code's correctness independently.
*   **Clear Communication:** The iterative debugging process highlighted the importance of clear communication, especially when dealing with environment-specific issues that require user interaction (e.g., restarting the server, checking console logs).
*   **Stubbing for Progress:** Stubbing the `BrewTruth` report allowed us to verify the API contract and integration points without waiting for the full implementation of the truth grading logic.

### 5. Conclusion

S4.9b.x is now functionally complete, with UI scaffolding, BrewTruth API integration, and robust handling of the `BREWTRUTH_ENABLED` flag in the development environment. The environment variable loading challenge was significant but ultimately resolved by a direct approach in `package.json`. The Jest tests provide confidence in the underlying logic. A follow-up task (S4.9b.env) has been documented to address the environment variable loading more robustly in the long term.
