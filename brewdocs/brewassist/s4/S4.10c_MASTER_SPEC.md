## S4.10c Master Spec

### Goal

Make **intent + toolbelt tier + BrewTruth** work as one consistent policy layer:

1.  **Intent resolved first** (what the user is trying to do)
2.  **Toolbelt tier resolved next** (what actions are allowed)
3.  **BrewTruth grades the response + flags risk** (how safe/trustworthy the outcome is)
4.  **Handshake decision**: allow / warn / block / require-confirmation

### Acceptance Criteria

You’re “green” on S4.10c when:

- BrewTruth is **not stubbed** and scores vary by content
- Flags populate (missing_context / speculative / possible_hallucination / safety_concern / fallback_used)
- Toolbelt “block” responses are explainable (“why blocked”, “what to do next”)
- Tests pass:
  - BrewTruth varies on different responses
  - High-risk content triggers safety flags and drops tier
  - Tool run truth check works and adds flags

## S4.10c.4 — BrewDocs Governance Capability (Tier-1)

This section details the introduction of BrewDocs as a first-class, governed capability within BrewAssist, leveraging the Unified Capability Registry. This enables BrewAssist to inspect, index, and reason about documentation without introducing drift or execution risk.

### Capability Definitions (Registry Truth)

The following capabilities have been added to `lib/capabilities/registry.ts`:

- **`brewdocs.inspect`**: Allows BrewAssist to inspect the local repository’s `brewdocs/` directory, adapt to the project’s domain, and report on its structure.
  - Category: `docs`
  - Tier: 1
  - Personas: `admin`, `dev`, `support`, `customer`
  - Intent: `DOCS_KB`
  - Confirm Required: No
  - Sandbox Required: No
  - RWX: Read
- **`brewdocs.read`**: Enables BrewAssist to read the content of documentation files within the `brewdocs/` directory.
  - Category: `docs`
  - Tier: 1
  - Personas: `admin`, `dev`, `support`, `customer`
  - Intent: `DOCS_KB`
  - Confirm Required: No
  - Sandbox Required: No
  - RWX: Read
- **`brewdocs.index`**: Facilitates the indexing of documentation locations, reporting missing or stale docs, and summarizing documentation coverage by module/phase.
  - Category: `docs`
  - Tier: 1
  - Personas: `admin`, `dev`, `support`, `customer`
  - Intent: `DOCS_KB`
  - Confirm Required: No
  - Sandbox Required: No
  - RWX: Read

### Command Surface: `/brewdocs` (Tier-1 Only)

The `/brewdocs` command surface is introduced with the following allowed actions:

- Detect `brewdocs/` presence.
- Dynamically adapt to project structure.
- Index documentation locations.
- Report missing or stale docs.
- Summarize documentation coverage by module/phase.

**Explicitly Forbidden Actions (Tier-1):**

- File writes
- Doc generation
- Cross-repo sync
- Embeddings

### Dynamic Structure Enforcement (Mandatory)

BrewAssist will inspect the local repository’s `brewdocs/` and adapt to the project’s domain, never assuming a static tree. The default adaptive template (guidance only) includes `project/`, `reference/`, `modules/`, `tasks/`, `tests/`, and `archive/`. A hard rule is enforced: never create module folders from another project domain.

### Non-Negotiables

- No hardcoded directory assumptions.
- No write access in Tier 1.
- No cross-project bleed.
- No sandbox execution in this phase.
- `lib/capabilities/registry.ts` is the single source of truth.

## Right Panel Cognition Surface

This section introduces a **Right-Side Cognition Panel** designed as a **non-executing DevOps intelligence surface**. Its primary purpose is to display high-level cognition states and contextualize DevOps 8 Principles, providing operational clarity without exposing internal chain-of-thought. This panel serves to prepare users for future BrewOps capabilities without implementing them yet.

### Cognition States

BrewAssist will surface the following state labels, not internal reasoning:

- Identity Check
- Context Awareness
- Scope Validation
- Risk Assessment
- Capability Matching
- Policy Enforcement
- Execution Planning
- Result Evaluation

**Hard Prohibitions:** No chain-of-thought, scoring math, internal prompts, model deliberation, or memory inspection will be exposed. This is for explainability, not transparency of internals.

### DevOps Principles Overlay (Educational)

The panel will display the following static DevOps 8 Principles, which may be highlighted contextually based on the task type (UI-only):

1.  Flow over batch
2.  Fast feedback
3.  Continuous learning
4.  Build quality in
5.  Limit work in progress
6.  Automate everything safe
7.  Make work visible
8.  Systems thinking

**Usage Rules:** These principles are read-only, involve no API calls, no persistence, no inference cost, and highlighting is purely UI-driven.

### Future BrewOps Compatibility Note

The right panel is designed as a future-compatible container. Currently, it functions as a single, scrollable panel with static content and dynamic labels, without tabs or integrations. It is architected to readily accommodate tabs, repo context, collaboration features, and ops intelligence in later phases without requiring significant refactoring.

**Explicit Future Note:**
“This cognition surface is intentionally non-interactive and non-integrated. It serves as the foundation for future BrewOps tab-based workspaces introduced in later phases.”
