## S4.10c Master Spec

### Goal

Make **intent + toolbelt tier + BrewTruth** work as one consistent policy layer:

1.  **Intent resolved first** (what the user is trying to do)
2.  **Toolbelt tier resolved next** (what actions are allowed)
3.  **BrewTruth grades the response + flags risk** (how safe/trustworthy the outcome is)
4.  **Handshake decision**: allow / warn / block / require-confirmation

### Acceptance Criteria

You’re “green” on S4.10c when:

*   BrewTruth is **not stubbed** and scores vary by content
*   Flags populate (missing_context / speculative / possible_hallucination / safety_concern / fallback_used)
*   Toolbelt “block” responses are explainable (“why blocked”, “what to do next”)
*   Tests pass:
    *   BrewTruth varies on different responses
    *   High-risk content triggers safety flags and drops tier
    *   Tool run truth check works and adds flags

## S4.10c.2 Acceptance Criteria (Flow Mode Output)

**Manual Validation Checklist:**

*   **Customer mode defaults ON:** Verify that in Customer mode, Flow Mode is enabled by default.
*   **Admin toggle works:** Verify that in Admin mode, the "Flow Mode ON/OFF" toggle correctly enables/disables Flow Mode.
*   **Skip works:** While an assistant message is progressively typing, click the "Skip" button and verify the full message content appears instantly.
*   **No "undefined" content renders:** Ensure no "undefined" or corrupted content appears during progressive rendering or after skipping.
*   **Final text equals original text:** Verify that the final rendered text of an assistant message exactly matches the original full text received from the API.
*   **Messages *feel spoken*:** Observe that messages "arrive" in a steady, word-based cadence.
*   **Chat doesn’t jump:** Confirm that if you scroll up, the chat does not auto-scroll back down until you return near the bottom.
*   **Markdown sections don’t have those giant gaps anymore:** Visually inspect markdown-rendered content for tighter spacing in paragraphs and lists.
## Scope Gate + Flow Mode

### Objective
1) Implement progressive response rendering (flow/typing) for chat output.
2) Enforce mode-aware scope gating so non-platform questions are handled correctly.

### Rules
- No chain-of-thought exposure.
- No new execution permissions.
- Must work in Admin and Customer modes.
- Must not break existing tests.

### Part 1 — Response Flow (UI)
- Implement a “Flow Mode” renderer that displays assistant responses progressively.
- Default ON in Customer mode, configurable in Admin.
- Provide a “Skip / Show all” control.
- Rendering should be word-based with light punctuation pauses.

### Part 2 — Scope Gate (Policy)
- Add an intent classification step BEFORE routing to LLM/HRM/Agent.
- Categories: PLATFORM_DEVOPS, SUPPORT, DOCS, GENERAL_KNOWLEDGE.
- Policy:
  - Customer mode: redirect GENERAL_KNOWLEDGE to platform help guidance.
  - Admin mode: allow GENERAL_KNOWLEDGE but label as “Non-Operational”.
- Ensure BrewTruth / cognition summary reflects scope category.

### Validation
- Add/extend a test proving Customer mode redirects GENERAL_KNOWLEDGE.
- Verify Flow Mode doesn’t alter final text output (only render behavior).
