# S4.9d — BrewAssist Toolbelt Reintegration

## Phase Name

**S4.9d — TOOLBELT_REINTEGRATION (Tiers + Modes + MCP Wizards)**

---

## Objective

Re-activate the BrewAssist Toolbelt (Tier 1 / Tier 2 / Tier 3) inside the **BrewAssist cockpit**, wired into:

- The **MCP Tools panel** (left sidebar),
- The **mode system** (HRM, LLM, Agent, Loop, Sandbox),
- And the emerging **Intent Gatekeeper + BrewTruth** pipeline.

The goal is to:

- Give vibe coders and devs **clear safety rails** around file / repo actions.
- Ensure high-risk operations (Toolbelt Tier 3) are **explicitly authorized**.
- Keep Tier 1 always available for safe analysis and guidance.

---

## Background

- Under BrewExec, the Toolbelt was defined in a markdown doc and partially wired into Gemini flows.
- BrewAssist moved to its own directory (`/brewassist`), leaving the Toolbelt **conceptual but not formally enforced** in the new UI + API stack.
- MCP tools (File Assistant, Git Command Center, Database Assistant, Research (NIMs), Suggest Edits) are now visible in the left sidebar but:
  - They lack **tier awareness**.
  - They do not yet reference a **central runtime config**.
  - They feel “always-on” rather than gated by experience and risk.

S4.9d will unify this into a **single source of truth**: `toolbeltConfig.ts`.

---

## Deliverables

1. **Runtime Config File**

   - New file: `lib/toolbeltConfig.ts`
   - Exposes:
     - Tier definitions (1 / 2 / 3)
     - Allowed actions per tier
     - Allowed modes per tier
     - MCP tool → minimum tier mapping
     - Human-readable labels and descriptions for UI

2. **Toolbelt → MCP Binding**

   - Left sidebar MCP buttons read `toolbeltConfig` before executing.
   - If the user’s **current tier** is insufficient:
     - Show a clear “Tier Required” message (no action taken).
   - If the action is Tier 2 or Tier 3:
     - Require confirmation or G.E.P. (Gemini Execution Protocol) where appropriate.

3. **Mode Awareness**

   - Toolbelt config expresses which tiers are allowed per mode:
     - HRM → Tier 1 only
     - LLM → Tier 1 & Tier 2
     - Agent → Tier 1, Tier 2, and supervised Tier 3
     - Loop → Tier 1 only
     - Sandbox → Tier 1 + constrained Tier 2
   - The UI displays safe/unsafe combinations (e.g., disabled buttons, tooltips).

4. **Minimal UI Feedback**

   - When a blocked action is attempted:
     - Non-modal, inline feedback: e.g., “Tier 3 change requires explicit approval.”
   - When a Tier 2 action is requested:
     - Modal preview with:
       - Files to be touched
       - Intention summary
       - “Apply / Cancel”

5. **Future Hook: Intent Gatekeeper + BrewTruth**

   - `toolbeltConfig` should include **optional hooks** for:
     - `requiresTruthCheck?: boolean`
     - `maxRiskLevel?: "low" | "medium" | "high"`
   - These fields will be honored in later phases (S5.x) when:
     - Intent gatekeeping,
     - BrewTruth scoring,
     - And override logic are fully wired.

---

## Non-Goals

- No actual MCP implementation refactor (Gemini can keep current handlers).
- No full Intent Gatekeeper integration yet.
- No BrewChat / BrewCore routing yet (those targets are still future).

This phase is **config and gating**, not a large behavior rewrite.

## Toolbelt Policy (Recap)

### Tier 1 — SAFE ACTIONS (Always Allowed)

- Read files, list directories
- Explain code
- Analyze logs
- Propose changes (no writes)
- Sandbox-only prototype (no disk writes)

**Modes:** HRM, LLM, Agent, Loop, Sandbox

---

### Tier 2 — GUIDED ACTIONS (Require Confirmation)

- Modify a single file
- Create files
- Delete obsolete files
- Small refactors and lint fixes
- Non-destructive shell commands

Requires:

1. Preview of affected files or diff
2. User confirmation (“Apply Patch”)

**Modes:** LLM, Agent, Sandbox (limited)

---

### Tier 3 — POWER CHANGES (Explicit Tier-3 Authorization)

- Multi-file refactors
- Layout rewrites
- Schema / config changes
- Cross-module rewiring
- Anything that could break the cockpit or repo

Requires:

1. User explicitly authorizes Tier 3.
2. G.E.P. header block (Gemini side).
3. File allow-list.
4. Dry-run / patch preview.
5. Final approval.

**Modes:** Agent (supervised only). Never in HRM / Loop.

### Step 2 — Wire MCP Sidebar to Toolbelt

- In `WorkspaceSidebarLeft.tsx` (or equivalent):
  - Before invoking any MCP handler, call:
    - `canRunTool(currentMode, currentTier, toolId)`

    - Show toast / inline message:

      - “Tier 3 change requires explicit Tier 3 authorization.”
  - If true:
    - Proceed with existing MCP flow.
