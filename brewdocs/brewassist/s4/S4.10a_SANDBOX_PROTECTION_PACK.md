# S4.10a — Sandbox Protection Pack

**Scope:** Lock the AI Sandbox to **Admin-only**, both in UI and API, and make sure Customer mode can’t even *see* it.

---

## 1. Objective

BrewAssist runs in two top-level cockpit modes:

* **Admin mode** – internal use only (RB + trusted devs).
* **Customer mode** – external users / clients.

The **AI Sandbox** is a raw, unfiltered execution space. It must be:

* **Visible and usable only in Admin mode**
* **Completely hidden + blocked in Customer mode**

This pack defines:

1. UI visibility rules
2. API / route guards
3. Logging + BrewLast hooks
4. Tests and acceptance criteria

---

## 2. Mode Source of Truth

Assumption: we either already have or are creating a mode context:

```ts
// contexts/CockpitModeContext.tsx
export type CockpitMode = 'admin' | 'customer';

export function useCockpitMode(): { mode: CockpitMode } { ... }
```

If the file already exists, just ensure:

* Values are `'admin'` and `'customer'` (not `dev` / `prod`)
* Default for local-dev is `'admin'`

All Sandbox protection logic must read from **this single source**.

---

## 3. UI Protection — Hide Sandbox in Customer Mode

### 3.1 Components to Guard

Expected components (names may vary slightly – Gemini can adapt):

* `components/AISandboxPanel.tsx` (or whatever renders the right-side sandbox UI)
* `components/AISandboxTrigger.tsx` / “Run Sandbox” button
* Any “BrewTruth Mode (cold audit)” toggle that’s Sandbox-specific

**Rule:**

```tsx
// In Sandbox panel
const { mode } = useCockpitMode();
if (mode === 'customer') return null;
```

Same pattern for any triggers:

```tsx
const { mode } = useCockpitMode();

if (mode === 'customer') {
  // No button, no icon, nothing
  return null;
}
```

### 3.2 Breadcrumb / HUD Clarity

In `BrewCockpitCenter.tsx` (or main header):

* When `mode === 'admin'`, show a tiny badge:
  `Admin Mode · Sandbox available`
* When `mode === 'customer'`, show:
  `Customer Mode · Sandbox locked (internal only)`

So if support ever screenshots the UI, it’s clear why Sandbox is missing.

## 4. API / Route Protection

Even if the UI hides Sandbox, users could try to:

* Call `/api/sandbox`
* Call `/api/brewassist` with `sandbox: true` or similar flags

### 4.1 Identify Sandbox Code Paths

Likely places:

* `pages/api/sandbox.ts` (dedicated route), **or**
* Extra flags in `pages/api/brewassist.ts` like `mode: 'sandbox'` or `raw: true`

Gemini should:

1. Search for “Sandbox” and “Run a raw prompt without BrewAssist tone…”
2. For each route or handler that runs Sandbox-like behavior, add a guard.

### 4.2 Minimal Guard (Phase 1)

Until we have full auth RBAC, use a simple **header-based mode hint** that the front-end sets:

```ts
// pages/api/sandbox.ts (or equivalent)
const cockpitMode = req.headers['x-brewassist-mode'];

if (cockpitMode !== 'admin') {
  return res.status(403).json({
    error: 'Sandbox is available only in Admin mode.',
    code: 'SANDBOX_FORBIDDEN',
  });
}
```

Front-end responsibility (in Sandbox client):

```ts
await fetch('/api/sandbox', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-brewassist-mode': mode, // 'admin' | 'customer'
  },
  body: JSON.stringify(payload),
});
```

> Later, S5.x can harden this to real auth+roles; for now this prevents casual misuse and keeps behavior consistent with the UI.

## 5. BrewLast & Logging

Every blocked Sandbox attempt should be logged so BrewAssist support can see what happened.

In a small helper, e.g. `lib/brewLast.ts` or `lib/brewInsightEngine.ts`:

```ts
export async function logSandboxBlocked(opts: {
  mode: string;
  path: string;
  reason: string;
}) {
  // For now: console + BrewLast entry
  console.warn('[SANDBOX_BLOCKED]', opts);

  await appendToBrewLast({
    type: 'sandbox_block',
    mode: opts.mode,
    path: opts.path,
    reason: opts.reason,
    at: new Date().toISOString(),
  });
}
```

Then in the API guard:

```ts
if (cockpitMode !== 'admin') {
  await logSandboxBlocked({
    mode: String(cockpitMode),
    path: req.url || '/api/sandbox',
    reason: 'non-admin attempted sandbox',
  });
  return res.status(403).json({ ... });
}
```

This feeds future **S5.6 Help/Support Engine** and incident analytics.

## 6. Tests

### 6.1 API Tests (Jest / Supertest style)

Create or extend:

* `__tests__/api/sandbox.protection.test.ts`

Scenarios:

1. **Admin allowed**

   * Send POST to `/api/sandbox` with header `x-brewassist-mode: admin`
   * Expect `status 200` (or happy-path status)

2. **Customer blocked**

   * Header `x-brewassist-mode: customer`
   * Expect `status 403` and `code: 'SANDBOX_FORBIDDEN'`

3. **Missing header blocked**

   * No mode header
   * Expect `status 403` and BrewLast log entry created

### 6.2 UI Tests (optional but nice)

In `__tests__/components/AISandboxPanel.test.tsx`:

* Render with context `mode='admin'` → Sandbox elements visible
* Render with context `mode='customer'` → Sandbox elements **not** in DOM
