# S4.9d.2 — Toolbelt Tier Rules → MCP Guard Rails

**Phase:** S4.9d.2
**Status:** Planned
**Depends on:** S4.9d.1 (HUD + Tier Selector online, `toolbeltConfig.ts` present)
**Goal:** Make the Toolbelt dropdown more than cosmetic by defining **exact rules** for what each tier may do, per **Mode** and **MCP Tool**.

---

## 1. Objectives

1.  Turn the existing Toolbelt dropdown (`Tier 1 – Safe`, `Tier 2 – Guided`, `Tier 3 – Power`) into a **governor** for all MCP actions.
2.  Define a **single source of truth** in `lib/toolbeltConfig.ts` that maps:
    *   `mode` (HRM / LLM / AGENT / LOOP)
    *   `tier` (T1 / T2 / T3)
    *   `toolId` (file, git, db, research, suggestEdits, etc.)
    → to **allowed / blocked** + **required confirmations**.
3.  Ensure **UI**, **Gemini CLI overlays**, and **API handlers** can all read the same rules.

---

## 2. Domain Model

### 2.1 Types (in `lib/toolbeltConfig.ts`)

```ts
// High-level power level
export type ToolbeltTier = 'T1_SAFE' | 'T2_GUIDED' | 'T3_POWER';

export type BrewModeId = 'HRM' | 'LLM' | 'AGENT' | 'LOOP';

// Current MCP tools (left column)
export type McpToolId =
  | 'file-assistant'
  | 'git-command-center'
  | 'database-assistant'
  | 'research-nims'
  | 'suggest-edits'
  | 'read-file'
  | 'list-directory'
  | 'explain-code'
  | 'propose-changes'
  | 'sandbox-prototype'
  | 'multi-file-refactors'
  | 'layout-rewrites'
  | 'schema-changes'
  | 'cross-module-rewiring'
  | 'break-repo'; // Added from existing ToolbeltToolId

export type ToolSafetyLevel =
  | 'read-only'          // inspect, list, explain only
  | 'single-file-write'  // create/modify/delete 1 file
  | 'multi-file-write';  // structural changes, multi-file patches

export interface ToolRule {
  enabled: boolean;
  safety: ToolSafetyLevel;
  // Does this action require an explicit “Yes, apply” confirmation?
  requireConfirmation: boolean;
  // Optional: require G.E.P. / Tier 3 header for this combination
  requireGepHeader?: boolean;
}

export type ToolbeltMatrix = {
  [mode in BrewModeId]: {
    [tier in ToolbeltTier]: {
      [tool in McpToolId]: ToolRule;
    };
  };
};
```

> **Note:** S4.9d.2 only defines the matrix; S4.9d.3 will enforce it.

---

## 3. Rules Matrix (Human Spec)

### 3.1 Mode vs Tier — Global Constraints

| Mode  | T1_SAFE                        | T2_GUIDED                                     | T3_POWER                                               |
| ----- | ------------------------------ | --------------------------------------------- | ------------------------------------------------------ |
| HRM   | Read-only only                 | ❌ (blocked)                                   | ❌ (blocked)                                            |
| LLM   | Read-only, suggestions only    | Single-file writes allowed with confirmation  | ❌ (blocked)                                            |
| AGENT | Read-only + single-file writes | Single-file writes + small multi-file patches | Full multi-file / structural changes (G.E.P. required) |
| LOOP  | Read-only only                 | ❌ (blocked)                                   | ❌ (blocked)                                            |

**Implications:**

*   **HRM** and **LOOP** are **never** allowed to write files or run destructive actions (Tier 2/3 → blocked).
*   **LLM** can write **only** under **T2 Guided** and only **one file at a time**.
*   **AGENT** is the **only mode** that can use **Tier 3 Power**.

---

### 3.2 MCP Tool Rules (per tool)

#### 3.2.1 `file-assistant`

*   **T1_SAFE**
    *   Allowed in HRM, LLM, AGENT, LOOP
    *   `safety: 'read-only'`
    *   Operations: list files, show content, explain code, propose patches (not applied)
*   **T2_GUIDED**
    *   Allowed in LLM, AGENT
    *   `safety: 'single-file-write'`, `requireConfirmation: true`
    *   Operations: create file, update single file, delete single file (if path explicit)
*   **T3_POWER**
    *   Allowed in AGENT only
    *   `safety: 'multi-file-write'`, `requireConfirmation: true`, `requireGepHeader: true`
    *   Operations: multi-file refactor, move/rename folders, bulk patch

#### 3.2.2 `git-command-center`

*   **T1_SAFE**
    *   Allowed in all modes
    *   `safety: 'read-only'`
    *   Operations: show git status, diff, log, branches
*   **T2_GUIDED**
    *   Allowed in LLM, AGENT
    *   `single-file-write`, `requireConfirmation: true`
    *   Operations: stage specific files, create commits with user-specified message
*   **T3_POWER**
    *   Allowed in AGENT only
    *   `multi-file-write`, `requireConfirmation: true`, `requireGepHeader: true`
    *   Operations: batch add/commit, branch operations, tag creation

#### 3.2.3 `database-assistant`

*   **T1_SAFE**
    *   Allowed in all modes
    *   Read-only DB introspection: list tables, schemas, generate migration suggestions
*   **T2_GUIDED**
    *   Allowed in LLM, AGENT
    *   Operations: generate migration scripts as text; apply only if user confirms
*   **T3_POWER**
    *   Allowed in AGENT only
    *   Execute multi-step migrations / data backfills with explicit G.E.P. header

#### 3.2.4 `research-nims`

*   **T1_SAFE**
    *   Allowed in all modes
    *   Pure web research + BrewTruth comparison, no file changes
*   **T2_GUIDED**
    *   Allowed in LLM, AGENT
    *   May propose patches or config updates, but application requires confirmation
*   **T3_POWER**
    *   **For now:** blocked (future phase for auto-apply NIMs-informed patches)

#### 3.2.5 `suggest-edits`

*   **T1_SAFE**
    *   Allowed everywhere; suggestions only, no auto-apply
*   **T2_GUIDED**
    *   Allowed in LLM, AGENT
    *   One-file patch apply with confirmation
*   **T3_POWER**
    *   Allowed in AGENT only, G.E.P. required for multi-file refactors

# S4.9d.3 — Toolbelt API Enforcement & Tests

**Phase:** S4.9d.3
**Status:** Planned
**Depends on:** S4.9d.1, S4.9d.2
**Goal:** Enforce toolbelt rules at the **API layer** so that no MCP action can violate Tier/Mode safeguards, and prove it via tests.

---

## 1. Objectives

1.  Use `getToolRule()` from `toolbeltConfig.ts` to **allow / block / require confirmation** for MCP actions.
2.  Ensure **HRM/LOOP cannot perform write actions**, **LLM cannot perform Tier 3**, and **AGENT Tier 3 always requires G.E.P. or explicit override**.
3.  Add Jest tests that simulate MCP calls at different tiers and modes, verifying correct behavior.

---

## 2. Enforcement Flow

### 2.1 Where to Enforce

*   Primary location: **`pages/api/brewassist.ts`** within the request handler, before executing any MCP action or file write.
*   Support helpers: `lib/brewassist-engine.ts` (for logging and structured error responses).

### 2.2 Request Contract

All MCP-related requests to `/api/brewassist` must now include:

```ts
type BrewAssistRequestPayload = {
  message: string;
  mode: BrewModeId;          // 'HRM' | 'LLM' | 'AGENT' | 'LOOP'
  toolbeltTier: ToolbeltTier; // 'T1_SAFE' | 'T2_GUIDED' | 'T3_POWER'
  mcpToolId?: McpToolId;     // e.g. 'file-assistant', optional for pure chat
  mcpAction?: string;        // e.g. 'readFile', 'writeFile', 'runMigration'
  confirmApply?: boolean;    // user clicked “Yes, apply patch”
  gepHeaderPresent?: boolean;// Gemini Execution Protocol header detected (Tier 3)
  // ...existing fields (context, sessionId, etc.)
};
```

> For non-MCP chat, `mcpToolId` can be undefined and enforcement becomes a no-op.

---

## 3. Enforcement Logic (Pseudocode)

In `/pages/api/brewassist.ts`:

```ts
import { getToolRule } from '@/lib/toolbeltConfig';

const payload = JSON.parse(req.body);
const { mode, toolbeltTier, mcpToolId, mcpAction, confirmApply, gepHeaderPresent } = payload;

if (mcpToolId) {
  const rule = getToolRule(mode, toolbeltTier, mcpToolId);

  if (!rule.enabled) {
    return res.status(403).json({
      error: 'TOOLBELT_FORBIDDEN',
      message: `This action is not allowed in ${mode} mode at ${toolbeltTier}.`,
    });
  }

  // Tier 2 / 3 confirmation
  if (rule.requireConfirmation && !confirmApply) {
    return res.status(409).json({
      error: 'TOOLBELT_CONFIRM_REQUIRED',
      message: 'Confirmation required before applying this change.',
    });
  }

  if (rule.requireGepHeader && !gepHeaderPresent) {
    return res.status(412).json({
      error: 'TOOLBELT_GEP_REQUIRED',
      message: 'Tier 3 actions require a valid G.E.P. header and explicit opt-in.',
    });
  }

  // Optional: enforce read-only vs write
  // e.g., if rule.safety === 'read-only' but mcpAction is writeFile → block
  if (rule.safety === 'read-only' && isWriteAction(mcpAction)) {
    return res.status(403).json({
      error: 'TOOLBELT_READ_ONLY',
      message: 'This tool is in read-only mode at the current tier.',
    });
  }
}

// If we reach here, call runBrewAssistEngine as usual
```

Add helper `isWriteAction(mcpAction: string): boolean` in a shared util.

## 4. Logging

In `lib/brewassist-engine.ts` (or within API route):

*   On **every blocked action**, log:

```ts
console.warn('[ToolbeltGuard]', {
  mode,
  toolbeltTier,
  mcpToolId,
  mcpAction,
  reason: errorCode, // e.g. TOOLBELT_FORBIDDEN
});
```

*   Consider attaching a small `toolbeltGuard` object into the BrewAssist response:

```ts
toolbeltGuard: {
  enforced: true;
  rule: { mode, toolbeltTier, mcpToolId, safety, requireConfirmation, requireGepHeader };
}
```

(This is optional for S4.9d.3 but useful later for BrewLast / audits.)

## 5. Test Plan

Create a new Jest file: `__tests__/api/brewassist.toolbelt.test.ts`.

### 5.1 Scenarios to Cover

1.  **HRM + T1_SAFE + file-assistant read** → **allowed**
    *   `mcpAction = 'readFile'` → 200 OK.
2.  **HRM + T2_GUIDED + file-assistant write** → **blocked**
    *   Expect 403 `TOOLBELT_FORBIDDEN`.
3.  **LLM + T1_SAFE + git-command-center commit** → **blocked**
    *   `mcpAction = 'commit'` → 403 `TOOLBELT_READ_ONLY`.
4.  **LLM + T2_GUIDED + file-assistant write, confirmApply=false** → **blocked**
    *   409 `TOOLBELT_CONFIRM_REQUIRED`.
5.  **LLM + T2_GUIDED + file-assistant write, confirmApply=true** → **allowed**.
6.  **AGENT + T3_POWER + file-assistant multi-file refactor, no G.E.P. header** → **blocked**
    *   412 `TOOLBELT_GEP_REQUIRED`.
7.  **AGENT + T3_POWER + file-assistant multi-file refactor, `gepHeaderPresent=true` & `confirmApply=true`** → **allowed**.
8.  **Loop + any Tier + any write action** → **blocked**.

### 5.2 Test Mechanics

*   Mock `global.fetch` or whatever is used internally so tests don’t hit real MCP endpoints.
*   Use minimal payloads, focusing on `mode`, `toolbeltTier`, `mcpToolId`, `mcpAction`, and flags.
*   Expect correct status codes and `error` codes in JSON.

---

*   [ ] HRM/LOOP never produce file or DB changes regardless of MCP tool selection.
*   [ ] LLM cannot execute Tier 3 actions; Tier 2 always requires confirmation.
*   [ ] AGENT Tier 3 requires **both** confirmation and G.E.P. header.
*   [ ] All new Jest tests in `brewassist.toolbelt.test.ts` pass.
*   [ ] Existing test suites remain green (no regressions).
*   [ ] When misconfigured, API returns **clear error codes** (not silent failures).
