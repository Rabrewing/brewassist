# S4.5 – BrewAssist Self-Repair Sandbox Architecture

**Project:** BrewExec / BrewAssist DevOps Cockpit
**Phase:** S4.5 – Self-Repair Sandbox Architecture
**Date:** 2025‑11‑26
**Owner:** RB + Production ChatG

---

## 1. Goal

The primary goal of S4.5 is to fully implement and integrate the Self-Repair Sandbox architecture, building upon the foundational self-maintenance engine introduced in S4.4. This phase will enable BrewAssist to generate, test, and manage code fixes within an isolated sandbox environment, ensuring that no production code is modified without explicit user approval.

---

## 2. Proposed Implementation

### 2.1 Dual-Repo Architecture

*   **Primary Repo (Production)**: `github.com/Rabrewing/brewexec` - This repository will continue to hold stable, production-ready code. BrewAssist will *never* write directly to this repo without explicit user approval.
*   **Sandbox Repo (AI DevOps)**: `github.com/Rabrewing/brewexec-sandbox` (or a special branch like `ai-dev`) - This will be the dedicated repository for all AI-generated files, patch plans, suggestions, and maintenance metadata. It mirrors the production repo but is safe to break and experiment with. All BrewAssist operations related to code generation and modification will default to this repo.

### 2.2 Sandbox Sync Logic

Implement logic to manage the synchronization and application of fixes within the sandbox:
1.  **File Existence Check**: Before proposing a fix for a file, BrewAssist will check if the file exists in the real (production) repo.
2.  **Copy to Sandbox**: If the file exists, it will be copied to the `sandbox/` directory (maintaining its relative path) before any modifications are made.
3.  **Sandbox-Only Modification**: BrewAssist will only modify the copy of the file within the `sandbox/` directory.
4.  **Status Marking**: The status of the proposed fix (e.g., "pending review", "applied to sandbox") will be marked in `.brewlast.json` or a dedicated maintenance log.
5.  **User Approval for Merge**: The merge process from the sandbox to the production repo will be initiated manually by RB via a command like `/approve_patch fix_003`.

### 2.3 Enhanced Self-Maintenance Engine

Expand the capabilities of `lib/brewassistMaintenance.ts` to:
*   **Detect Broken Tools**: Implement more sophisticated checks to identify non-functional or misconfigured tools.
*   **Detect Miswired Routes**: Add logic to verify API route configurations.
*   **Detect Missing Files**: Enhance checks for critical file presence.
*   **Suggest Self-Repairs**: Improve the `generateMaintenanceSuggestions` function to propose concrete code fixes for detected issues.
*   **Generate Code Fixes**: Integrate with the LLM to generate actual code changes for identified problems, writing these changes to the `sandbox/` directory using `pages/api/brewassist-sandbox-apply.ts`.

---

## 3. Acceptance Criteria (To be defined)

Specific acceptance criteria will focus on verifying:
*   BrewAssist can detect a simulated broken file in the production repo.
*   BrewAssist can generate a code fix for the broken file.
*   The generated fix is written *only* to the `sandbox/` directory.
*   The production repo remains untouched by AI-generated fixes without explicit approval.
*   The `/approve_patch` command (or similar mechanism) can be used by RB to merge a sandbox fix into production.
