1. Purpose

S4.10 formalizes BrewAssist as a dual-surface platform with one engine and two operating modes:

Admin Mode — Internal, full DevOps cockpit

Customer Mode — External, safe, proposal-driven experience

This separation is required to:

Prevent privilege leakage

Eliminate accidental execution

Enable SaaS readiness

Preserve DevOps credibility

Create a clean upgrade path without re-architecture

2. Mode Definitions

2.1 Admin Mode (Internal)

Admin Mode is unrestricted.

Admin Mode unlocks:

Sandbox panel

MCP tools

Action Menu (full)

Toolbelt Tier 1–3 execution

Chain engines (LLM / HRM / Agent / Loop)

Wizards (/init, file ops, patch ops)

BrewTruth (visible, interactive)

Gatekeeper override

Debug logs

BrewLast persistence

Experimental panels

Admin UI Behavior

Full cockpit layout

Risk labels visible

TruthChip visible

Context chips visible

Advanced shortcuts enabled

2.2 Customer Mode (External)

Customer Mode is safe, clean, and non-destructive.

Customer Mode hides:

Sandbox

File tree previews

MCP tools

Action Menu

Wizards

Repo/file execution

Debug surfaces

Dev logs

Mode toggle

Customer UI Behavior

Clean assistant-only interface

Formatting toolbar only

No internal metadata

No dev affordances

7. Access Control & Enforcement Layers

All privileged actions pass through four aligned gates:

A) ToolbeltGuard

Enforces tier + mode

Blocks execution for customers

B) Intent Gatekeeper

Determines allowed intent per mode

Customer cannot escalate privilege

C) Sandbox Protection

Sandbox never visible or callable by customers

D) API Guards

All privileged APIs must enforce:

if (mode === "customer" && request.isDestructive) { return 403; }
